5 ## Introduction
6 
7 **EnigmaIoT** is an open source solution for wireless multi sensor systems. It has two main components, multiple **nodes** and one **gateway**.
8 
9 A number of nodes with one or more sensors each one communicate in a **secure** way to a central gateway in a star network using EnigmaIoT protocol.
10 
11 This protocol has been designed with security on mind. All sensor data is encrypted with a random key that changes periodically. Key is unique for each node and dynamically negotiated, so user do not have to enter any key. Indeed, all encryption and key agreement is transparent to user.
12 
13 I designed this because I was searching for a way to have a relatively high number of nodes at home. I thought about using WiFi but it would overload my home router. So I looked for an alternative. I evaluated LoRa or cheap nRF24 modules but I wanted the simplest solution in terms of hardware.
14 
15 ESP8266 microcontroller implements a protocol known as ESP-NOW. It is a point to point protocol, based on vendor specific [WiFi management action frames](https://mrncciew.com/2014/09/29/cwap-802-11-mgmt-frame-types/), that works in a connectionless way and every packet is a few milliseconds long. Because of this it eases to have a battery powered node so that it enables designing totally wireless sensors.
16 
17 But use of encryption on ESP-NOW limits the number of nodes to only 6 nodes. So I thought that I could implement encryption on payload but I found many problems I should solve to grade this as "secure enough".
18 
19 Find library documentation on https://gmag11.github.io/EnigmaIOT

21 ## Project requirements
22 
23 During this project conception I decided that it should fulfil this list of requirements.
24 
25 - Use the simplest hardware, based on ESP8266.
26 - Secure by design. Make use of a secure channel for data transmission.
27 - Automatic dynamic key agreement.
28 - Do not require connection to the Internet.
29 - Do not overload my home WiFi infrastructure. Only Gateway may be connected to LAN.
30 - Able to use deep sleep to run on batteries.
31 - Enough wireless range for a house.
32 - Support for a high number of nodes.

34 ## Features
35 
36 - [x] Encrypted communication using [**ChaCha20/Poly1305**](https://tools.ietf.org/html/rfc7539)
37 - [x] Dynamic key, shared between one node and gateway. Keys are independent for each node
38 - [x] Shared keys are expired after a certain (configurable) time
39 - [x] Number of nodes is only limited by memory on gateway (60 bytes per node)
40 - [x] Key is never on air so it is cannot be captured
41 - [x] Key expiration and renewal is managed transparently
42 - [x] Avoid repeatability attack having a new random initialization vector on every message. This is mandatory for ChaCha20/Poly1305 in order to keep cipher secure
43 - [x] Automatic and transparent node attachment
44 - [x] Avoid rogue node, rogue gateway and man-in-the-middle attack
45 
46 Notice that network key used to implement this feature is stored on flash. ESP8266 do not allow flash encryption so network key may be recovered reading flash. On the other side, ESP32 is able to encrypt flash memory, but EnigmaIoT is not still tested on it.
47 
48 - [x] Pluggable physical layer communication. Right now only ESP-NOW protocol is developed but you can easily add more communication alternatives
49 - [x] When using ESP-NOW only esp8266 is needed. No more electronics apart from sensor
50 - [x] Data message counter to detect lost or repeated messages
51 - [x] Designed as two libraries (one for gateway, one for node) for easier use
52 - [x] Crypto algorithm could be changed with low effort
53 - [x] Node and Gateway do store shared keys only on RAM. They are lost on power cycle. This protects system against flash reading attack. All nodes attach automatically with a new shared key after gateway is switched on
54 - [x] Downlink available. If deep sleep is used on sensor nodes, it is queued and sent just after node send a data message
55 - [x] Optional sleep mode management. In this case key has to be stored temporally. Normally RTC memory is the recommended place, and it is the one currently implemented
56 - [x] Initial configuration over WiFi portal on each device
57 - [x] Node configuration while in service using control downlink commands
58 - [ ] OTA over WiFi
59 - [x] OTA over MQTT/ESP-NOW
60 - [x] Sensor identification by using a flashing LED. This is useful when you have a bunch of nodes together :D
61 - [ ] Broadcast messages that go to all nodes
62 - [ ] Node grouping to send messages to several nodes at once

64 ## Design
65 
66 ### System Design
67 
68 System functions are divided in three layers: application, link and physical layer.
69 
70 ![Software Layers](https://github.com/gmag11/EnigmaIOT/raw/master/img/system_layers.png)
71 
72 - **Application layer** is not controlled by EnigmaIoT protocol but main program. User may choose whatever data format or final destination of payload. A good option is to use CayenneLPP format but any other format or even raw data may be used. The only limit is the maximum packet length that, for ESP-NOW is around 200 bytes.
73 
74 - **Link layer** is the one that add privacy and security. It manages connection between nodes and gateway in a transparent way. It does key agreement and node registration and checks the correctness of data messages. In case of any error it automatically start a new registration process. On this layer, data packets are encrypted using calculated symmetric key.
75 
76 - **Physical layer** currently uses connectionless ESP-NOW. But a hardware abstraction layer has been designed so it is possible to develop interfaces for any other layer 1 technology like LoRa or nRF24F01 radios.
77 
78 ### EnigmaIoT protocol
79 
80 The named **EnigmaIoT protocol** is designed to use encrypted communication without the need to hardcode the key. It uses [Elliptic Curves Diffie Hellman](https://en.wikipedia.org/wiki/Elliptic-curve_Diffieâ€“Hellman) algorithm to calculate a shared key.
81 
82 The process starts with node announcing itself with a Client Hello message. It tells the gateway its intention to establish a new shared key. It sends public key to be used on gateway to calculate the shared key.
83 
84 Gateway answers with Server Hello message that includes its public key for shared key calculation on node.
85 
86 Once shared key is calculated, node send an encrypted message as Key Exchange Finished message. Poly1305 encryption tag is used to check message integrity.
87 
88 If gateway validates tag correctly it answers with a Cipher Finished message.
89 
90 This process is protected with a 32 byte shared **network key**, used for **authentication**. As in the first two messages shared key is not known yet, Client Hello and Server Hello messages are encrypted whit this network key. If network key is not the same on gateway and node this will lead to decryption errors and messages will be ignored.
91 
92 In case of any error on node key negotiation gateway sends an Invalidate Key to reset to original status and forgets any calculated shared key for this node.
93 
94 When key is marked as valid node may start sending sensor data.
95 
96 Optionally, gateway can send data to node. As node may be sleeping between communications, downlink messages has to be sent just after uplink data. So, one downlink message is queued until node communicates. Node waits some milliseconds before sleep for downlink data.
97 
98 If a new downlink message arrives, old scheduled data for that node, if any, is overwritten.
99 
100 In case of nodes that do not sleep (like a mains powered relay), gateway can send downlink data in any moment. Sleepy node is signaled during node registration on a bit in Key Exchange Finished message. It is set to 1 to signal that node will sleep just after sending data.
101 
102 Key is forced to change every period. Gateway decides the moment to invalidate each node key. If so, it sends an invalidate key as downlink, after next data message communication. This key validity period is configurable on [EnigmaIoTconfig.h](https://github.com/gmag11/EnigmaIOT/blob/master/src/lib/EnigmaIoTconfig.h) file.
103 
104 After that node may start new key agreement sending a new Client Hello message.
105 
106 All nodes and gateway are identified by its MAC address. No name is assigned so no configuration is needed on node. Function assignment has to be done at a higher level.

114 ## Message format specification
115 
116 ### Client Hello message
117 
118 ![Client Hello message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/ClientHello.png)
119 
120 Client hello is sent by node to start registration procedure. It includes the public key to be used on Elliptic Curve Diffie Hellman (EDCH) algorithm to calculate the key. Initialization vector (IV) is used for encryption. There is a random 4 byte field reserved for future use.
121 
122 This message is sent encrypted with network key.
123 
124 ### Server Hello message
125 
126 ![Server Hello message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/ServerHello.png)
127 
128 After receiving and checking Client Hello message, gateway responds with a Server Hello message. It carries gateway's public key to let node calculate key using ECDH. There is a random 4 byte field reserved for future use. Gateway assigns node a NodeID. It is signaled as a 2 byte field.
129 
130 Server Hello message is sent encrypted with network key.
131 
132 ### Sensor Data message
133 
134 ![Node payload message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/SensorData.png)
135 
136 Sensor data is always encrypted using shared key and IV. Apart from payload this message includes node ID and a counter used by gateway to check lost or repeated messages from that node.
137 
138 Total message length (without tag) is included on a 2 byte field.
139 
140 ### Node control message (downlink)
141 
142 ![Node control message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/SensorCommand-Downlink.png)
143 
144 Gateway can send commands to an individual node in a similar way as sensor data is sent by nodes. For nodes that can be slept between consecutive data messages this commands are queued and sent just after a data message is received.
145 
146 Only last message is queued. In case Gateway tries to send a new message, old one gets deleted and overridden by the new one.
147 
148 ### Control message
149 
150 ##### Downlink Control Message
151 
152 ![DL Control Command message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/ControlComand-Downlink.png)
153 
154 ##### Uplink Control Message
155 
156 ![UL Control Command message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/ControlComand-Uplink.png)
157 
158 Gateway and node can exchange internal control commands. These are used to set internal protocol parameters like sleep time. This type of messages are processed like normal downlink messages, but are not passed to higher level (user code) in Node.
159 
160 Some control messages, like OTA update messages, require that they are processed immediately. Hence, it is required that node is not in deep sleep mode. This can be controlled, for instance, using another control command to set sleep time to 0.
161 
162 ### Invalidate Key message
163 
164 ![Invalidate Key message format](https://github.com/gmag11/EnigmaIOT/raw/master/img/InvalidateKey.png)
165 
166 After every data message from nodes, gateway evaluates key integrity and validity. In case of any error decoding the packet gateways ignores data and reply with this message indicating the reason that caused error. Node must start a new registration procedure in order to send data again. After this new registration node resends the last data message.
167 
168 A gateway defines a key validity period after that a node key is marked as expired. In a message is received after that is processed normally but an Invalidate Key message indicating key expiration as reason. Node then starts a registration procedure but does not retry communication.
169 
170 Invalidate Key message is always sent unencrypted.

208 ## Data format
209 
210 Although it is not mandatory at all, use of [CayenneLPP format](https://mydevices.com/cayenne/docs/lora/#lora-cayenne-low-power-payload) is recommended for sensor data compactness.
211 
212 You may use [CayenneLPP library](https://github.com/ElectronicCats/CayenneLPP) for encoding on node and decoding on Gateway.
213 
214 Example gateway code expands data message to JSON data, to be used easily as payload on a MQTT publish message to a broker. For JSON generation [ArduinoJSON](https://arduinojson.org) library is required.
215 
216 In any case you can use your own format or even raw unencoded data. Take care of maximum message length that communications layer uses. For ESP-NOW, maximum payload length it is 217 bytes.

218 ## Output data from gateway
219 
220 ### Uplink messages
221 
222 A user may program their own output format modifying gateway example program. For my use case gateway outputs MQTT messages in this format:
223 ```
224 <configurable prefix>/<node address>/sensordata <json data>
225 ```
226 A prefix is configured on gateway to allow several sensor networks to coexist in the same subnet. After that address and data are sent.
227 
228 After every received message, gateway detects if any packet has been lost before and reports it using MQTT message using this format:
229 ```
230 <configurable prefix>/<node address>/status {"per":<packet error rate>,"lostmessages":<Number of lost messages>,"totalmessages":<Total number of messages>,"packetshour":<Packet rate>}
231 ```
232 If ESP-NOW is used for node communication, ESP8266 cannot use WiFi in a reliable way. That's why in this case gateway will be formed by two ESP8266 linked by serial port. First one will output MQTT data in form of ascii strings and second one will forward them to MQTT broker.
233 
234 ### Downlink messages
235 
236 EnigmaIoT allows sending messages from gateway to nodes. In my implementation I use MQTT to trigger downlink messages too.
237 
238 To make it simpler, downlink messages use the same structure than uplink.
239 ```
240 <configurable prefix>/<node address>/<command> <data>
241 ```
242 Node address means destination node address. Configurable prefix is the same used for uplink communication.
243 
244 Commands are developed by user, but some are reserved for control commands. An uplink message could be like this
245 
246 ```
247 enigmaiot/12:34:56:78:90:12/light ON
248 ```
249

311 ## OTA Update
312 
313 OTA updates are transferred using the same mechanism. Firmware is sent over MQTT using a [Python script](./EnigmaIoTUpdate/EnigmaIoTUpdate.py). Then gateway selects the appropriate node and send this binary data over ESP-NOW.
314 
315 As ESP-NOW restricts **maximum payload to 250 bytes per message** firmware is splitted in chunks. Every chunk is **212 bytes** long, so that it fits together with message headers and is multiple of 4. This splitting work is done by `EnigmaIoTUpdate.py` script.
316 
317 ### Using EnigmaIoTUpdate.py
318 
319 A requirement is to have installed [Python3](https://www.python.org/download/releases/3.0/) in the computer used to do the update.
320 
321 In order to run the update, you need to install [`paho-mqtt`](https://pypi.org/project/paho-mqtt/) library. To do that you can follow instructions [here](https://pypi.org/project/paho-mqtt/#installation).
322 
323 ```
324 $python3 ./EnigmaIoTUpdate.py --help
325 
326 Usage: EnigmaIoTUpdate.py [options]
327 
328 Options:
329  -h, --help show this help message and exit
330  -f FILENAME, --file=FILENAME
331  File to program into device
332  -d ADDRESS, --daddress=ADDRESS
333  Device address
334  -t BASETOPIC, --topic=BASETOPIC
335  Base topic for MQTT messages
336  -u MQTTUSER, --user=MQTTUSER
337  MQTT server username
338  -P MQTTPASS, --password=MQTTPASS
339  MQTT server user password
340  -S MQTTSERVER, --server=MQTTSERVER
341  MQTT server address or name
342  -p MQTTPORT, --port=MQTTPORT
343  MQTT server port
344  -s, --secure Use secure TLS in MQTT connection. Normally you should
345  use port 8883
346  --unsecure Use secure plain TCP in MQTT connection. Normally you
347  should use port 1883
348 ```
349 
350 An example of this command could be like this:
351 
352 ```
353 python3 ./EnigmaIoTUpdate.py \
354  -f EnigmaIOTsensor.bin \
355  -d 11:22:33:44:55:66 \
356  -t enigmaiot \
357  -u "mymqttbrokeruser" \
358  -P "mymqttbrokerpassword" \
359  -S mysecure.mqtt.server \
360  -p 8883 \
361  -s
362 ```
363 
364 Notice that using ESP-NOW device address correspond to **MAC address** of your ESP8266.
365 
366 It is very important to configure user and password on you MQTT broker. Besides, if it is going to be accessed from the Internet you should activate TLS encryption and a certificate.
